SpringMVC-文档搬运工 STOMP

简介

WebSocket是一种全双工通信，而SockJS为它提供了浏览器兼容(某些浏览器不支持WebSocket，所以SockJS为它们提供回退，自动降为轮询)。实现服务器与客户端之间自由通信，服务器可主动地推送消息到客户端，而不是传统的请求-响应模型
而STOMP就是传输的简单数据协议

STOMP原理与过程

  • 首先在服务器端通过Config类注册前缀(app目的地前缀、简单中继器前缀和升级WebSocket请求)
    • 这些前缀是告诉服务器如何处理这些请求
    • app目的地前缀：通过Controller@MesageMapping或@SubscribeMapping最终转发到中继器或者变成点对点通信，一般是转发到消息中继器(亦可以说是Spring Boot内部中间件的简单实现)
    • 简单的中继器前缀：符合中继器前缀的请求都进入中继器然后去掉前缀，看匹配上那个路由键，就触发中继器发布订阅信息
    • 升级WebSockety请求：升级过程完全封装，无需理会，只需定义。其过程就是客户端发送的请求头中带有Upgrade: websocket，Connection: Upgrade等发起WebSocket连接
  • 然后在前端JS可以定义订阅那个频道，当然需要加上中继器前缀
  • 当客户端发送带有中继器前端的请求，就会进入到中继器，匹配上路由键，中间可能夹杂一些业务处理(比如拍卖的的出价的入队等)，然后中继器会发布订阅信息到所有订阅了该频道的客户端(比如当前实时出价的更新)
  • 而当客户端发送戴欧app目的地请求的前缀时，会根据拿掉前缀后的请求匹配上那个Hadnler，再转发到中继器，然后匹配路由键，发布订阅信息
    其实说白了，WebScoket的全双工通信就是在HTTP的基础上在客户端与服务端之间加多了一个虚拟层，就是消息队列或者叫消息中间件(中继器)，它接收来自服务端或者客户端的信息(带有进入它的前缀),然后匹配信息应该发送到哪个频道。由它来做了主动发送的动作。本质上它是一个转发消息的队列，所以实现发布订阅非常的简单，也会它的原理之一。

概述

简单讲一下SpringMVC的异步请求原理和一些个人的理解

DeferredResult

Callable

异步的中间结果
java.util.concurrent.Callable

异步的处理过程

  • 可以通过调用request.startAsync()将ServletRequest置于异步模式。这样做的主要作用是Servlet（以及任何过滤器）可以退出，但响应保持打开状态，以便稍后处理完成
  • 对request.startAsync()的调用返回AsyncContext，可以使用它来进一步控制异步处理。例如，它提供了dispatch方法，类似于Servlet API中的forward，除了它允许应用程序在Servlet容器线程上恢复请求处理
  • ServletRequest提供对当前DispatcherType的访问，您可以使用它来区分处理初始请求，异步调度，转发和其他调度程序类型
    DeferredResult处理过程：
  • controller返回DeferredResult并把它保存在内存中
  • SpringMVC调用request.startAsync()
  • 同时DispatcherServlet和所有拦截器退出请求处理器线程，当响应保持打开状态
  • 应用从某线程中设置DeferredResult，SpringMVC调度请求返回到Servlet容器
  • DispatcherServlet在此被调用，处理异步产生的值
    Callable处理过程：
  • controller返回Callable
  • SpringMVC调用request.startAsync()，把Callable提交到TaskExecutor以便在单独的线程中处理
  • 同时DispatcherServlet和所有拦截器退出请求处理器线程，当响应保持打开状态
  • 当Callable产生一个结果，SpringMVC调度请求返回到Servlet容器
  • DispatcherServlet再次被调用，处理异步返回的值
    总结：
    实际上就是通过一个未处理完成的中间结果把从容器中抽出来放到别的线程中去继续处理，而此时容器可以继续响应其他请求而达到异步的结果。其中相应结果保持打开，让真正结果处理出来时可以得到响应。当结果出来时，容器优先响应

    异步的异常处理

    使用DeferredResult时，可以选择是否使用异常调用setResult或setErrorResult。在这两种情况下，Spring MVC都会将请求发送回Servlet容器以完成处理。然后将其视为控制器方法返回给定值或者就好像它产生了给定的异常一样。然后异常通过常规异常处理机制（例如，调用@ExceptionHandler方法）

但是在Spring Boot的默认集成中并不需要那么麻烦。
实际上Spring Boot在配置依赖的时候已经有了非常多的默认配置在properties上(一般皆是在本地服务器上部署的默认配置，可通过IDE查看。即个人开发基本上不需要再配置什么)。

Context Hierarchy(上下文层次)

DispatcherServlet容易包含着两大上下文

  • Servlet WebApplicationContext：负责创建Controller，视图解析器(ViewResolver)，映射处理器(HandlerMapping)
  • Root WebapplicationContext：负责创建Service层、Repository层
    在Spring Boot中亦无需配置其映射

    Special Bean Types

  • GandlerMapping 负责将请求映射到拦截器(预处理、后处理)
  • HandlerAdapter 负责协助DispatcherServlet决定最终调用哪个Handler(用于DispatcherServlet解耦)
  • HandlerExceptionResolver 解析异常并把异常映射到合适的处理器、HTML错误页面或其他
  • ViewResolver 视图解析器，主要把Controller的返回值解析成视图再返回到响应
  • LocaleResolver 语言环境解析器(国际化),根据时区等提供国际化视图环境
  • ThemeResolver 主题解析器，应用哪个视图主题比如提供个人的布局
  • MultipartResolver 解析由多种类型组成的请求例如上传，识别content携带的数据类型
  • FlashMapManager 略

    Web MVC Config

    Spring Boot提供主要JavaConfiguration来进行MVC配置
    配置Servlet：
    AbstractAnnotationConfigDispatcherServletInitialize实现WebApplicationInitializer，是更简单配置的抽象基础类，通过重写相关方法进行配置

Spring Boot无需配置，除非有特殊的个人配置
一般少量个性化可在properties里配置

Processing(SpringMVC的处理请求过程)

  • WebApplicationContext搜索处理过程中可以使用的Controller和其他元素作为属性绑定进请求当中
  • 绑定环境解析器(locale resolver)，允许处理请求时解析需要使用的环境设置
  • 绑定主题解析器(theme resolver)，允许请求内元素(视图决定使用哪个主题)可使用主题
  • 请求被多类型解析器(multiparts)检查(如果指定了默认的多类型解析器)。若请求中有类被匹配上，则请求被包装成MultipartHttpServletRequest被其他元素(multipart handling)进行更深入的处理
  • 搜寻合适的Handler(泛指于预处理器，后处理器，Controllers)。以此执行处理链，准备model或render(对于被注解的Controller就是视图)
  • 若返回了model，那就呈现view。若没有返回model(可能被预处理或后处理拦截，或者处于安全问题)，那么view也不会呈现，因为请求可能已经完成了
    声明在WebApplicationContext里的HandlerExceptionResolver是用来解析在请求处理过程中出现的异常。允许自定义
    通过增加Servlet initialization的参数来定义个性化的DispatcherServlet
    DispatcherServlet initialization parameters
  • contextClass 需要实现ConfigurableWebApplicationContext接口。默认使用XmlWebApplicationContext
  • contextConfigLocation 传递给上下文实例（由contextClass指定）的字符串，用于指示可以在何处找到上下文
  • namespace
  • throwExceptionIfNoHandlerFound 是否抛出没有适合Handler的处理的异常。默认为False，因为可以设置返回404而不用抛出异常

    拦截器(Interception)

    Handler拦截器
  • preHandle(..) 在handler之前
  • postHandle(..) 在hanler之后
  • afterCompletion(..) 请求结束之后
    preHandle返回的是boolean值，ture表示可以继续在处理链上处理下去，false表示不再继续下去，在preHandle已完成
    注意：postHandle对于@ResponseBody和@ResponseEntity方法不太有用，在HandlerAdapter和postHandle之前，响应已经被写入并提交。

    Exceptions

    请求映射或handler抛出异常时，DispatcherServlet会委派HandlerExceptionResolver Beans去解析异常和提供解决方案，这就是错误响应
    HandlerExceptionResolver implementations
  • SimpleMappingExceptionResolver 映射异常到错误视图(根据名字)，对返回错误页面到浏览器比较有用
  • DefaultHandlerExceptionResolver 解析由Spring MVC引发的异常并映射到HTTP状态码上。可用ResponseEntityExceptionHandler替代
  • ResponseStatusExceptionResolver 使用@ResponseStatus annotation注解映射异常到HTTP状态码，取决于注解中的设定的值(哪个状态码)
  • ExceptionHandlerExceptionResolver 通过调用@Controller或@ControllerAdvice中的@ExceptionHandler方法去解析异常

    解析器链(Chain of Resolvers)

    通过声明多种HandlerExceptionResolver beans并排序在Spring Configuration去构成异常解析链。数字越大，位置越后
    HandlerExceptionResolver约定返回值：

RabbitMQ

RabbitMQ介绍

RabbitMQ是实现AMQP（高级消息队列协议）的消息中间件的一种
AMQP，即Advanced Message Queuing Protocol，高级消息队列协议，是应用层协议的一个开放标准，为面向消息的中间件设计。消息中间件主要用于组件之间的解耦，消息的发送者无需知道消息使用者的存在，反之亦然。AMQP的主要特征是面向消息、队列、路由（包括点对点和发布/订阅）、可靠性、安全
RabbitMQ是一个开源的AMQP实现，服务器端用Erlang语言编写，支持多种客户端，如：Python、Ruby、.NET、Java、JMS、C、PHP、ActionScript、XMPP、STOMP等，支持AJAX。用于在分布式系统中存储转发消息，在易用性、扩展性、高可用性等方面表现不俗
RabitMq在消息队列上加入一层抽象：交换机。用于让生产者与队列没有直接联系，而是给交换机，交换机再把消息加入到队列中去。

概念

RabbitMQ每个服务器有虚拟主机、交换机、队列和绑定

  • 虚拟主机：一个虚拟主机持有一组交换机、队列和绑定。RabbitMQ中永驻只能以虚拟主机的粒度进行权限控制，所以一个一台服务器拥有多个虚拟主机。如果要禁止A组访问B组的交换机/队列/绑定。必须为A和B分别创建一个虚拟主机。
  • 交换机：用于转发信息，不做储存。如果没有 Queue bind 到 Exchange 的话，它会直接丢弃掉 Producer 发送过来的消息。根据路由键来决定发到哪个队列中去，路由键位于消息中
  • 绑定：交换机和队列绑定

    交换机

    四种类型交换机
  • Direct：direct 类型的行为是”先匹配, 再投送”. 即在绑定时设定一个 routing_key, 消息的routing_key 匹配时, 才会被交换器投送到绑定的队列中去
  • Topic：按规则转发消息（最灵活）
  • Headers：设置header attribute参数类型的交换机
  • Fanout：转发消息到所有绑定队列

    Direct Exchange

    Direct Exchange是RabbitMQ默认的交换机模式，也是最简单的模式，根据key全文匹配去寻找队列
    当消息中的路由键与交换机中的binding key全文匹配上时，才会转发到对应的队列中

    Topic Exchange

    转发消息主要是根据通配符
    在这种交换机模式下：
  • 路由键必须是一串字符，用句号（.） 隔开，比如说 agreements.us，或者 agreements.eu.stockholm 等
  • 路由模式必须包含一个 星号（*），主要用于匹配路由键指定位置的一个单词，比如说，一个路由模式是这样子：agreements..b.*，那么就只能匹配路由键是这样子的：第一个单词是 agreements，第三个单词是 b。 井号（#）就表示相当于一个或者多个单词，例如一个匹配模式是agreements.eu.berlin.#，那么，以agreements.eu.berlin开头的路由键都是可以的
    topic 和 direct 类似, 只是匹配上支持了”模式”, 在”点分”的 routing_key 形式中, 可以使用两个通配符:
  • *表示一个词
  • #表示零个或多个词

    Headers Exchange

    headers 也是根据规则匹配, 相较于 direct 和 topic 固定地使用 routing_key , headers 则是一个自定义匹配规则的类型. 在队列与交换器绑定时, 会设定一组键值对规则, 消息中也包括一组键值对( headers 属性), 当这些键值对有一对, 或全部匹配时, 消息被投送到对应队列

    Fanout Exchange

    Fanout Exchange 消息广播的模式，不管路由键或者是路由模式，会把消息发给绑定给它的全部队列，如果配置了routing_key会被忽略

environments 配置环境

MyBatis 可以配置成适应多种环境，这种机制有助于将 SQL 映射应用于多种数据库之中， 现实情况下有多种理由需要这么做，例如，开发、测试和生产环境需要有不同的配置
如果你想连接两个数据库，就需要创建两个 SqlSessionFactory 实例，每个数据库对应一个。而如果是三个数据库，就需要三个实例

  • 默认的环境 ID（比如:default=”development”）
  • 每个 environment 元素定义的环境 ID（比如:id=”development”）
  • 事务管理器的配置（比如:type=”JDBC”）(Spring + MyBatis，则没有必要配置事务管理器， 因为 Spring 模块会使用自带的管理器来覆盖前面的配置)
  • 数据源的配置（比如:type=”POOLED”）

    数据源（dataSource）

    使用标准的 JDBC 数据源接口来配置 JDBC 连接对象的资源
    有三种内建的数据源类型（也就是 type=”[UNPOOLED|POOLED|JNDI]”）
  • UNPOOLED 这个数据源的实现只是每次被请求时打开和关闭连接
  • POOLED 这种数据源的实现利用“池”的概念将 JDBC 连接对象组织起来，避免了创建新的连接实例时所必需的初始化和认证时间。 这是一种使得并发 Web 应用快速响应请求的流行处理方式
  • JNDI 这个数据源的实现是为了能在如 EJB 或应用服务器这类容器中使用

  • resultSets 这个设置仅对多结果集的情况适用，它将列出语句执行后返回的结果集并每个结果集给一个名称，名称是逗号分隔的

自动映射

正如你在前面一节看到的，在简单的场景下，MyBatis可以替你自动映射查询结果。 如果遇到复杂的场景，你需要构建一个result map。 但是在本节你将看到，你也可以混合使用这两种策略。 让我们到深一点的层面上看看自动映射是怎样工作的
当自动映射查询结果时，MyBatis会获取sql返回的列名并在java类中查找相同名字的属性（忽略大小写）。 这意味着如果Mybatis发现了ID列和id属性，Mybatis会将ID的值赋给id
通常数据库列使用大写单词命名，单词间用下划线分隔；而java属性一般遵循驼峰命名法。 为了在这两种命名方式之间启用自动映射，需要将 mapUnderscoreToCamelCase设置为true
自动映射甚至在特定的result map下也能工作。在这种情况下，对于每一个result map,所有的ResultSet提供的列， 如果没有被手工映射，则将被自动映射。自动映射处理完毕后手工映射才会被处理。 在接下来的例子中， id 和 userName列将被自动映射， hashed_password 列将根据配置映射

有三种自动映射等级：

  • NONE 禁用自动映射。仅设置手动映射属性
  • PARTIAL 将自动映射结果除了那些有内部定义内嵌结果映射的(joins)
  • FULL 自动映射所有
    默认值是PARTIAL

    这个简单语句的效果如下:
  • 映射语句文件中的所有 select 语句将会被缓存
  • 映射语句文件中的所有 insert,update 和 delete 语句会刷新缓存
  • 缓存会使用 Least Recently Used(LRU,最近最少使用的)算法来收回
  • 根据时间表(比如 no Flush Interval,没有刷新间隔), 缓存不会以任何时间顺序 来刷新
  • 缓存会存储列表集合或对象(无论查询方法返回什么)的 1024 个引用
  • 缓存会被视为是 read/write(可读/可写)的缓存,意味着对象检索不是共享的,而 且可以安全地被调用者修改,而不干扰其他调用者或线程所做的潜在修改

这个更高级的配置创建了一个 FIFO 缓存,并每隔 60 秒刷新,存数结果对象或列表的 512 个引用,而且返回的对象被认为是只读的,因此在不同线程中的调用者之间修改它们会 导致冲突
可用的收回策略有:

  • LRU 最近最少使用的:移除最长时间不被使用的对象
  • FIFO 先进先出:按对象进入缓存的顺序来移除它们
  • SOFT 软引用:移除基于垃圾回收器状态和软引用规则的对象
  • 弱引用:更积极地移除基于垃圾收集器状态和弱引用规则的对象
    默认的是 LRU
    flushInterval(刷新间隔)可以被设置为任意的正整数,而且它们代表一个合理的毫秒 形式的时间段。默认情况是不设置,也就是没有刷新间隔,缓存仅仅调用语句时刷新
    size(引用数目)可以被设置为任意正整数,要记住你缓存的对象数目和你运行环境的 可用内存资源数目。默认值是 1024
    readOnly(只读)属性可以被设置为 true 或 false。只读的缓存会给所有调用者返回缓 存对象的相同实例。因此这些对象不能被修改。这提供了很重要的性能优势。可读写的缓存 会返回缓存对象的拷贝(通过序列化) 。这会慢一些,但是安全,因此默认是 false
